# 05. 專題實作整合指南 (Project Integration)

本單元將 Agile/Scrum 流程對應到學校學期制的專題開發時程。

## 1. 學期時程規劃範例 (Semester Timeline)

假設一學期約 16-18 週，扣除期中考與期末考，我們約有 12-14 週的開發時間。建議切分為 **6-7 個 Sprints (每個 Sprint 2 週)**。

### Phase 1: 探索與準備 (第 1-2 週)
*   **活動**: 組隊、題目發想、建立 GitHub Repo、架設開發環境。
*   **產出**:
    *   初步的 Product Backlog (約 10-20 個粗略的故事)。
    *   第一版架構圖。
    *   Hello World 專案已可執行。

### Phase 2: 核心功能衝刺 (Sprint 1-4, 第 3-10 週)
這是開發的黃金期。
*   **Sprint 1**: 完成最基礎架構 (登入、資料庫連線、共用版型)。**目標：可以 Demo 的骨架。**
*   **Sprint 2**: 核心功能 A (例如：商品列表 + 購物車)。
*   **Sprint 3**: 核心功能 B (例如：結帳流程 + 訂單查詢)。
*   **Sprint 4**: 核心功能 C (例如：後台管理)。
*   **重點**: 每個 Sprint 結束都要能展示！不要等到期中考才第一次跑程式。

### Phase 3: 收尾與優化 (Sprint 5-6, 第 11-14 週)
*   **Sprint 5**: 次要功能 (例如：搜尋、篩選、會員中心修飾) + 修復 Bug。
*   **Sprint 6**: UI/UX 美化、RWD 調整、簡報製作、部署上線。

---

## 2. 如何應對「評分」與「老師需求」？

學校專題通常會有固定的審查點（Proposal, 期中, 期末）。

*   **利用 Sprint Review 練習報告**: 每次 Sprint 結束的 Demo，就是期末報告的預演。
*   **將「製作簡報」視為一個 User Story**: 把「製作期中報告投影片」也寫成一張卡片放入 Sprint Backlog，因為它也會花時間。
*   **面對老師的「需求變更」**:
    *   老師：「我覺得這裡要加一個 AI 推薦功能。」
    *   傳統反應：「天啊，要重做，做不完！」
    *   **敏捷反應**：「好的老師，我們把它加入 Product Backlog。但因為時間有限，我們會評估它的優先順序，可能會排擠掉原本『會員積分』的功能，這樣可以嗎？」(這展現了你們對專案管理的專業度)。

## 3. 常見失敗模式與解法

1.  **Git 衝突地獄**:
    *   **解法**: 任務切得夠細 (Small Stories)，每個人做不同的檔案；頻繁 Commit 與 Pull。
2.  **只有一個人在寫程式**:
    *   **解法**: 結對程式設計 (Pair Programming)。強的人帶弱的人，不要讓強者包辦，否則他生病專題就掛了。
3.  **最後一週才整合**:
    *   **解法**: 堅持每個 Sprint 都要 Merge 到主分支 (Master/Main) 並且要是能動的。

## 4. 結語

敏捷不是萬靈丹，它不會幫你寫程式，但它像是一個導航系統：**它會讓你盡早知道你走錯路了**，讓你還有時間修正。

> **"Fail Fast, Learn Faster." (盡早失敗，更快學習)**

祝各位專題開發順利！
