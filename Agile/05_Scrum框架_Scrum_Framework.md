# 03. Scrum 框架實戰 (Scrum Framework)

Scrum 是一個輕量級的框架，幫助團隊解決複雜問題。對於學生專題，Scrum 能讓你們保持固定的開發節奏，避免「最後一週才趕工」。

## 1. Scrum 的三個角色 (Roles)

在學生專題中，你們可能需要一人分飾多角，但職責要分清楚：

1.  **Product Owner (PO / 產品負責人)**:
    *   **職責**: 決定「做什麼」。負責維護 Backlog，決定功能優先順序。
    *   **專題對應**: 通常由對專題方向最清楚的組長擔任，或是全組共同決策（但要有人最後拍板）。
2.  **Scrum Master (SM / 敏捷教練)**:
    *   **職責**: 確保流程順暢，排除障礙，舉辦會議。不是「老闆」，而是「僕人式領導」。
    *   **專題對應**: 負責提醒大家開會、追蹤進度的人。
3.  **Developers (開發團隊)**:
    *   **職責**: 實際動手做 (前端、後端、設計、測試)。
    *   **專題對應**: 全體組員。

## 2. Scrum 的五個事件 (Events)

以一個 **Sprint (衝刺)** 為週期，建議學生專題設定為 **2週**。

1.  **Sprint Planning (衝刺規劃會議)**:
    *   **時間**: Sprint 第一天。
    *   **內容**: 從 Product Backlog 選出這兩週要做什麼故事？如何做？
2.  **Daily Scrum (每日站立會議)**:
    *   **時間**: 每天 15 分鐘 (學生可改為每週 2-3 次實體，其餘線上)。
    *   **內容**: 昨天做了什麼？今天打算做什麼？有沒有遇到困難？
3.  **Sprint Review (檢視會議/Demo)**:
    *   **時間**: Sprint 最後一天。
    *   **內容**: **展示 (Demo)** 做好的功能給老師或組員看。不要只報告投影片，要跑程式！
4.  **Sprint Retrospective (回顧會議)**:
    *   **時間**: Review 之後。
    *   **內容**: 檢討「流程」。我們這次哪裡做得好？哪裡可以改進？
    *   **推薦工具：KPT 法**:
        *   **Keep (保持)**: 做得好的事，下次繼續。（例：每天都有準時站會）
        *   **Problem (問題)**: 遇到的困難或做得不好的。（例：GitHub 衝突太多）
        *   **Try (嘗試)**: 針對 Problem，下個 Sprint 要嘗試的改進。（例：大家每 2 小時 Push 一次程式碼）

## 3. 專題實作練習 (Homework)

### 任務：啟動你們的第一個 Sprint
1.  **角色分配**: 確認誰是 PO，誰是 SM。
2.  **定義 Sprint 長度**: 建議定為 2 週。
3.  **規劃會議**: 挑選出接下來 2 週能完成的 User Stories，放入 **Sprint Backlog**。
4.  **開始開發**: 進入下一個模組，使用看板來管理這些工作。
